"How do you sequence schema changes for large tables?"

A strong answer is:

For small tables, simple ALTER statements are usually sufficient. For large tables, I use phased deployments: add nullable columns first, backfill data in batches, create indexes online, validate data, then enable constraints and application code changes. I always deploy parent objects before child objects, database changes before dependent procedures, and prepare rollback scripts for every release. This minimizes locking, redo/undo generation, and production downtime.

"What are FIRST_CHANGE# and NEXT_CHANGE# in V$LOG?"

FIRST_CHANGE# is the starting SCN contained in a redo log file, and NEXT_CHANGE# is the SCN where that log ends. Together they define the SCN range covered by a redo log sequence. Oracle uses these values during crash recovery, media recovery, RMAN restores, and Data Guard synchronization to determine which redo logs contain the required changes. For the CURRENT log, NEXT_CHANGE# is set to a very large value because Oracle is still writing to that log.

Q: After migration, new inserts fail with duplicate key violations. What do you check?

Answer:

The first thing I check is whether the target sequence is synchronized with the maximum primary key value in the table. During migrations, data is often loaded with explicit IDs, but the sequence does not automatically advance. I compare MAX(id) with the sequence's current value and use setval() to move the sequence beyond the highest existing key before enabling application writes.

Interview Answer

During data migrations, I migrate schema objects first, then load data, validate counts and integrity, and finally synchronize sequences or identity columns. A common issue is that sequences do not automatically advance when rows are loaded with explicit primary keys. To prevent duplicate key errors after cutover, I compare the sequence value with MAX(id) and use PostgreSQL's setval() function to align the sequence. For large migrations, I use tools such as AWS Database Migration Service or Ora2Pg and perform validation checks before switching production traffic.

how to get process id and how to identify active and inactive process ids in database and how to identify sql stmts which is currently running and taking long times

This is a very common PostgreSQL DBA interview question.

1. Get All PostgreSQL Process IDs (PIDs)

Use pg_stat_activity:

SELECT pid,
       usename,
       datname,
       application_name,
       state,
       query
FROM pg_stat_activity;

Example output:

pid	usename	state	query
12345	postgres	active	SELECT * FROM orders
12346	appuser	idle	COMMIT
12347	appuser	idle in transaction	BEGIN
2. Identify Active Sessions
SELECT pid,
       usename,
       datname,
       state,
       query_start,
       query
FROM pg_stat_activity
WHERE state = 'active';

States:

State	Meaning
active	Currently executing SQL
idle	Connected but doing nothing
idle in transaction	Transaction open, waiting
disabled	Stats collection disabled
3. Identify Inactive Sessions
SELECT pid,
       usename,
       state,
       backend_start
FROM pg_stat_activity
WHERE state = 'idle';

These sessions consume connections but are not executing SQL.

4. Find Long-Running Queries

One of the most important DBA queries:

SELECT pid,
       usename,
       now() - query_start AS runtime,
       state,
       query
FROM pg_stat_activity
WHERE state = 'active'
ORDER BY runtime DESC;

Example:

pid	runtime
54321	01:15:22
54322	00:20:15

Shows the longest-running queries first.

5. Find Queries Running More Than 5 Minutes
SELECT pid,
       usename,
       now() - query_start AS runtime,
       query
FROM pg_stat_activity
WHERE state = 'active'
  AND now() - query_start > interval '5 minutes';
6. Find Idle Transactions (Dangerous)

These can hold locks and prevent VACUUM.

SELECT pid,
       usename,
       now() - xact_start AS txn_age,
       query
FROM pg_stat_activity
WHERE state = 'idle in transaction';

Example:

BEGIN;
UPDATE orders SET amount=100;
-- user forgot COMMIT

Transaction remains open.

7. Find Blocking Sessions
SELECT blocked.pid AS blocked_pid,
       blocked.query AS blocked_query,
       blocking.pid AS blocking_pid,
       blocking.query AS blocking_query
FROM pg_stat_activity blocked
JOIN pg_stat_activity blocking
ON blocking.pid = ANY(pg_blocking_pids(blocked.pid));

Example:

Blocked PID : 12345
Blocking PID: 12340
8. Find Locks
SELECT pid,
       locktype,
       mode,
       granted
FROM pg_locks;

Useful for diagnosing:

Deadlocks
Blocking
Long waits
9. Find Sessions Waiting on Something
SELECT pid,
       usename,
       wait_event_type,
       wait_event,
       state,
       query
FROM pg_stat_activity
WHERE wait_event IS NOT NULL;

Example:

wait_event_type	wait_event
Lock	transactionid
IO	DataFileRead
Client	ClientRead
10. Find CPU/Resource-Heavy Queries (pg_stat_statements)

Enable extension:

CREATE EXTENSION pg_stat_statements;

Then:

SELECT query,
       calls,
       total_exec_time,
       mean_exec_time
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 10;

Shows:

Most expensive SQL
Frequently executed SQL
Average execution time

This is one of the first places a PostgreSQL DBA looks for performance tuning.

11. Kill a Query

Cancel query only:

SELECT pg_cancel_backend(12345);
12. Terminate Session

Kills connection:

SELECT pg_terminate_backend(12345);

Be careful—this rolls back any active transaction.

In real-world database design, the question "How do you handle large volumes of data and BLOBs?" is often testing your understanding of scalability, storage, performance, and architecture.

Interview Answer

For enterprise applications, I generally avoid storing large BLOBs directly in PostgreSQL unless the files are small and frequently accessed. Large files such as images, PDFs, videos, and documents are typically stored in object storage like Amazon S3, while PostgreSQL stores only metadata and file references. This approach keeps the database size manageable, improves backup and recovery times, reduces replication overhead, and provides better scalability. For structured data in the TB range, I use partitioning, read replicas, and archiving strategies. For very large datasets, I consider sharding or data lake architectures.

Q: Difference between CHAR and VARCHAR?

CHAR is a fixed-length datatype and pads unused space with blanks, while VARCHAR is variable length and stores only actual characters entered. CHAR is suitable for fixed-size values like country codes, whereas VARCHAR is preferred for names, emails, and descriptions because it saves storage and avoids unnecessary padding.

Q: What is NVL?

NVL is an Oracle function used to replace NULL values with a specified value. In PostgreSQL, the equivalent is COALESCE, and in MySQL, the equivalent is IFNULL.

Q: What are constraints?

Constraints enforce data integrity. Common constraints are PRIMARY KEY, FOREIGN KEY, UNIQUE, NOT NULL, CHECK, and DEFAULT. They ensure valid, consistent, and reliable data in the database.

Sharding vs Read Replicas vs Partitioning
Technique	Purpose
Read Replica	Scale reads
Partitioning	Optimize large tables in one DB
Sharding	Scale writes + data size horizontally
12. Interview Answer

Sharding is a horizontal scaling technique where data is split across multiple independent database instances. Each shard holds a subset of the data based on a sharding key such as customer_id, region, or hash value. Unlike partitioning, which operates within a single database, sharding distributes data across multiple servers and requires application-level routing. It helps scale write-heavy and large datasets but introduces challenges like cross-shard queries, rebalancing, and global consistency. PostgreSQL does not natively support sharding, but it can be implemented using application logic or extensions like Citus.


PostgreSQL partitioning is a technique to split large tables into smaller physical segments based on a key such as range, list, or hash. It improves performance through partition pruning, reduces maintenance overhead, and makes large datasets easier to manage. Range partitioning is commonly used for time-series data, list partitioning for categorical data like region, and hash partitioning for even distribution. Each partition can have its own indexes, and queries automatically scan only relevant partitions. However, poor partition key selection or too many partitions can degrade performance.

Oracle provides advanced partitioning capabilities including range, list, hash, composite, interval, and reference partitioning with automatic partition creation and multi-level subpartitioning.
